**Props** (short for "properties") and **State** are the two core data mechanisms in React that determine what a component renders. While both hold data that influences the UI, they have entirely different purposes and data-flow rules.

---

### What are Props?

Props are inputs you pass from a parent component down to a child component. They are **read-only (immutable)** from the perspective of the receiving component.

Think of props like function arguments: a parent component calls a child component and hands it a set of parameters to display.

```jsx
// Parent Component passes props
export default function App() {
  return <UserProfile username="Alice" role="Admin" />;
}

// Child Component receives props
function UserProfile({ username, role }) {
  // ❌ ERROR: You cannot modify props!
  // username = "Bob";

  return (
    <div>
      {username} is an {role}
    </div>
  );
}
```

---

### Props vs. State: The Key Differences

| Feature             | Props                                                                         | State                                                                                        |
| ------------------- | ----------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| **Origin**          | Passed down from a **parent** component.                                      | Managed **internally** by the component itself.                                              |
| **Mutability**      | **Immutable** (Read-only). A child cannot change its own props.               | **Mutable** (Changeable) via state setter functions (e.g., `useState`).                      |
| **Primary Purpose** | To configure a child component from the outside and share data down the tree. | To handle internal data that changes over time (like form inputs, toggles, or fetched data). |
| **Analogy**         | Function arguments passed into a recipe.                                      | The changing ingredients inside the mixing bowl as you bake.                                 |

---

### How They Work Together

A common pattern in React is **lifting state up** or flowing data down:

1. The **parent** component holds the dynamic **state** (e.g., `const [count, setCount] = useState(0)`).
2. The parent passes that state down to a child component as a **prop** (e.g., `<CounterDisplay count="{count}"/>`).
3. The child component simply receives the prop and renders it. If the user interacts with the app, an action triggers a state change in the parent, which recalculates and passes new props down.

Based on the React documentation's guidelines for **"Choosing the state structure"**, writing clean state is about avoiding redundant data, preventing synchronization bugs, and structuring your state so it is impossible to enter invalid UI states.

Here is a detailed breakdown of the core principles from the docs, complete with code examples showing the **wrong way** vs. the **right way**.

---

## 1. Avoid Redundant State (Derived State)

**The Rule:** If you can compute a value from existing props or state during rendering, **do not put it into state**.

Putting derived data into state creates synchronization bugs because you now have to manually update two different pieces of state whenever data changes.

### ❌ The Wrong Way (Redundant State)

```jsx
import { useState } from "react";

export default function Form() {
  const [firstName, setFirstName] = useState("Jane");
  const [lastName, setLastName] = useState("Doe");

  // ❌ Redundant! Storing computed state in a separate variable using useEffect
  // or duplicate handlers is an anti-pattern.
  const [fullName, setFullName] = useState("Jane Doe");

  function handleFirstNameChange(e) {
    setFirstName(e.target.value);
    setFullName(e.target.value + " " + lastName); // Easy to forget or get out of sync
  }

  return <p>{fullName}</p>;
}
```

### ✅ The Right Way (Compute During Render)

Calculate `fullName` directly on the fly. It is always perfectly in sync and requires zero extra state hooks.

```jsx
import { useState } from "react";

export default function Form() {
  const [firstName, setFirstName] = useState("Jane");
  const [lastName, setLastName] = useState("Doe");

  // ✅ Computed during render. Always accurate!
  const fullName = `${firstName} ${lastName}`;

  return (
    <div>
      <input value={firstName} onChange={(e) => setFirstName(e.target.value)} />
      <input value={lastName} onChange={(e) => setLastName(e.target.value)} />
      <p>Full name: {fullName}</p>
    </div>
  );
}
```

## 2. Avoid Duplication in State

**The Rule:** If the exact same data is mirrored across multiple state variables, or inside an object _and_ an array, your state is duplicated.

### ❌ The Wrong Way (Duplicated Data)

Imagine a selection menu where clicking an item stores the _entire item object_ in state, but you also have a separate array of all items. If an item's details change elsewhere, your selected item becomes stale.

```jsx
export default function Menu() {
  const [items, setItems] = useState([
    { id: 1, name: "Burger" },
    { id: 2, name: "Pizza" },
  ]);

  // ❌ Duplicating data: storing the whole object instead of just the ID
  const [selectedItem, setSelectedItem] = useState({ id: 1, name: "Burger" });
}
```

### ✅ The Right Way (Store IDs, Not Objects)

Store only the unique identifier (`id`) in state, and look up the full item data from your master list when you need it.

```jsx
import { useState } from "react";

export default function Menu() {
  const [items, setItems] = useState([
    { id: 1, name: "Burger" },
    { id: 2, name: "Pizza" },
  ]);

  // ✅ Store only the ID. If the item updates, selectedId is never stale.
  const [selectedId, setSelectedId] = useState(1);

  const selectedItem = items.find((item) => item.id === selectedId);

  return (
    <ul>
      {items.map((item) => (
        <li key={item.id} onClick={() => setSelectedId(item.id)}>
          {item.name} {item.id === selectedId && "(Selected)"}
        </li>
      ))}
    </ul>
  );
}
```

## 3. Avoid Deeply Nested State

**The Rule:** If your state is nested deeply inside multiple objects, it becomes rigid, hard to update immutably, and prone to mutation bugs. Flat data structures are much easier to manage.

### ❌ The Wrong Way (Deep Nesting)

Trying to update a deeply nested property requires tedious spread syntax (`...`) at every level.

```jsx
// Hard to update safely
const [travelPlan, setTravelPlan] = useState({
  id: 0,
  title: "(Root)",
  childPlaces: [
    {
      id: 1,
      title: "Planet Earth",
      childPlaces: [{ id: 2, title: "Americas" }],
    },
  ],
});
```

### ✅ The Right Way (Normalized State / Flat Tables)

Flatten your state structure by using IDs as keys pointing to flat records, similar to how a database stores relational data.

```jsx
// Flat and easy to look up and update by ID
const [places, setPlaces] = useState({
  0: { id: 0, title: "(Root)", childIds: [1] },
  1: { id: 1, title: "Planet Earth", childIds: [2] },
  2: { id: 2, title: "Americas", childIds: [] },
});
```

In React, setting state does not immediately change the variable in your current code snapshot. Instead, it **requests a render** and builds an internal **update queue**.

Understanding how React queues and batches these updates is crucial for avoiding bugs when a state change depends on a previous one.

---

## 1. The Core Concept: State as a Snapshot & Batching

When you call a state setter like `setNumber(number + 1)`, the `number` variable represents a fixed snapshot for that specific render.

React **batches** state updates, meaning it waits until all code in your event handler has finished running before it processes the queue and triggers a single re-render.

### The Trap: Why Multiple Direct Setters Fail

If you try to increment a number three times in a single click event using the direct value:

```javascript
const [number, setNumber] = useState(0);

function handleClick() {
  setNumber(number + 1); // setNumber(0 + 1)
  setNumber(number + 1); // setNumber(0 + 1)
  setNumber(number + 1); // setNumber(0 + 1)
}
```

**What happens under the hood:**

1. `number` is `0` throughout the entire event handler.
2. React adds three separate instructions to its queue: _"Replace state with `1`"_.
3. React batches them together, processes the queue, and finalizes the state at `1` instead of `3`.

---

## 2. The Solution: Passing an Updater Function

If you want to update the same state variable multiple times before the next render, you can pass an **updater function** (e.g., `n => n + 1`) to the state setter instead of a raw value.

```javascript
const [number, setNumber] = useState(0);

function handleClick() {
  setNumber((n) => n + 1); // Passes 0 -> returns 1
  setNumber((n) => n + 1); // Passes 1 -> returns 2
  setNumber((n) => n + 1); // Passes 2 -> returns 3
}
```

### How React Processes the Queue with Updaters:

- **Step 1:** React adds the first function `n => n + 1` to its queue.
- **Step 2:** React adds the second function `n => n + 1` to its queue.
- **Step 3:** React adds the third function `n => n + 1` to its queue.
- **Step 4:** During the next render, React loops through the queue. It takes the previous state (`0`), passes it to the first updater (getting `1`), passes _that_ result to the second updater (getting `2`), and finally passes it to the third (getting `3`).

---

## 3. Mixing Values and Updaters

You can even mix regular values and updater functions in the same queue. React executes them sequentially:

```javascript
const [number, setNumber] = useState(0);

function handleClick() {
  setNumber(number + 5); // Replaces state with 5 (assuming initial is 0)
  setNumber((n) => n + 1); // Takes 5, adds 1 -> returns 6
  setNumber(42); // Replaces state outright with 42
}
```

In this scenario, React processes the queue step-by-step and ultimately sets the state to **`42`**.
